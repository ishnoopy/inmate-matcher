# Scraper Service

This package contains county-specific scrapers that produce normalized inmate roster JSON and local mugshot files for downstream matching.

## Location

- Limestone scraper: `src/limestone.ts`
- Madison scraper: `src/madison.ts`
- Shared helpers: `utils/index.ts`
- Output rosters: `data/rosters/*.json`
- Output photos: `data/photos/<source>/*.jpg`

## Run

From `apps/scraper`:

```bash
pnpm run scrape:limestone
pnpm run scrape:madison
```

From repo root:

```bash
pnpm -C apps/scraper run scrape:limestone
pnpm -C apps/scraper run scrape:madison
```

## Output Contract

Each scraper emits `RosterEntry` records from `@inmate-matcher/shared` with:

- `id`: stable source-prefixed hash ID (`limestone:<sha1>` or `madison:<sha1>`)
- `source`: county source key
- `fullNameRaw` and `nameNormalized`
- `bookingNumber` (Madison) or `null`
- `dob` (Limestone when available) or `null`
- `photoUrls`: relative file paths under `data/photos`
- `scrapedAt`: ISO timestamp

`nameNormalized` is generated by `normalizeName()` and stored in uppercase
(example: `SMITH JOHN ALLEN`).

Both scripts deduplicate using `Map<string, RosterEntry>` keyed by `id`, then sort by `nameNormalized` before writing JSON.

## Limestone (`src/limestone.ts`)

### How it works

1. Calls a JSON POST API endpoint:
   - `https://limestone-al-911.zuercherportal.com/api/portal/inmates/load`
2. Uses server-side paging (`count=50`, `start=<offset>`) until all records are fetched.
3. Builds a stable ID from:
   - `name | dob | arrest_date`
4. Decodes base64 mugshots from the API payload and stores them as:
   - `data/photos/limestone/<sha1>.jpg`
5. Writes final roster to:
   - `data/rosters/limestone.json`

### Reliability choices

- `postJson()` wraps API calls with retry logic (default 3 attempts, 1s delay).
- A short pause (`sleep(100)`) is added between page requests.
- Mugshot save failures are logged but do not abort the whole scrape.

## Madison (`src/madison.ts`)

### How it works

1. Uses Playwright (`chromium`) to scrape roster pages from:
   - `https://www.madisoncountysheriffal.org/inmate-roster`
2. Bootstraps by opening the roster and detecting last page from the `Last` pagination link.
3. Splits page numbers across parallel browser workers.
4. For each roster card, extracts:
   - profile URL
   - mugshot URL
   - name
   - booking number
5. Builds a stable ID from:
   - `fullNameRaw | bookingNumber | profileUrl`
6. Downloads mugshots to:
   - `data/photos/madison/<sha1>.jpg`
7. Writes final roster to:
   - `data/rosters/madison.json`

### Concurrency and controls

`src/madison.ts` reads environment variables (see `.env`):

- `SCRAPER_HEADLESS` (`true` by default)
- `SCRAPER_SLOWMO_MS` (`0` by default in code)
- `SCRAPER_CONCURRENCY` (`4` by default in code, clamped to page count)

Photo downloads are deduplicated with `pendingPhotoDownloads` so concurrent workers do not download the same file twice.

## Shared Utilities (`utils/index.ts`)

- `ensureDirExists(dir)`: creates output directories recursively.
- `sha1(input)`: deterministic IDs and filenames.
- `sleep(ms)`: pacing helper.
- `getPageNumberFromHref(href, base)`: supports query-param and path-based pagination formats.
- `resolveUrl(href, base)`: converts relative links to absolute URLs.

## Notes and Tradeoffs

- The Limestone scraper is API-first and generally more stable/faster.
- The Madison scraper is DOM-dependent and can break if selectors or page structure change.
- Both scrapers are intentionally tolerant of photo download issues so roster JSON still gets produced.
